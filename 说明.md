# TicLib 项目架构说明

本文档面向后续接手 TicLib 底层开发的人，说明当前项目架构、TiC/ConArm 工具盔甲的底层模型、已有模块职责，以及新增功能时应遵守的开发规范。

`docs/Crt.md` 是 CraftTweaker 和 Java API 用法文档；本文档只描述现有架构和开发约束。

## 1. 项目定位

TicLib 是一个面向 TiC 1.12.2 / Construct's Armory 的底层辅助库。

项目目标：

- 统一识别 TiC 工具和 ConArm 盔甲。
- 统一读取和写入工具/盔甲 root NBT。
- 统一处理 build、modify、rebuild 后的状态重放。
- 统一接入 Forge capability、CraftTweaker build event 和玩家运行时缓存。
- 给外部 Java 和 CT 脚本提供稳定 facade。

外部 Java 代码和 CT 集成层应优先依赖：

```text
com.smd.ticlib.api.*
```

## 2. TiC/ConArm 底层模型

在 TiC 1.12.2 / ConArm 中，工具和盔甲的核心形态是：

```text
ItemStack
  root NBT
    Base
    Stats
    Traits
    Modifiers
    Materials
```

TiC/ConArm 会根据材料、base modifier、modifier NBT 和构建事件重新计算物品最终状态。TicLib 在这个模型上增加一层私有组件数据，用于保存库自己的持久状态。

当前数据边界：

- TiC 原生数据：`Stats`、`Traits`、`Modifiers`、`Base.Modifiers`、材料列表。
- TicLib 持久数据：`root.ticlib.Components.<moduleId>`。
- 运行时缓存：只存在内存中，例如玩家盔甲 trait 快照。

TicLib 私有数据结构：

```text
root.ticlib.DataVersion
root.ticlib.DirtyVersion
root.ticlib.Components.<moduleId>
```

当前内置组件：

```text
root.ticlib.Components["ticlib:stats"]
root.ticlib.Components["ticlib:fluid"]
```

开发约束：

- 库状态只写入 `root.ticlib.Components`。
- TiC 原生 trait/modifier 只用于表达 TiC 原生语义。
- token、流体、自定义模块进度、运行时同步标记等数据不要写入 `Traits`、`Modifiers`、`Base.Modifiers`。

## 3. 当前分层

项目当前分为三层：

```text
core    底层能力
module  具体功能
api     外部 facade
```

### 3.1 core

`core` 只处理底层通用能力，不包含具体业务模块。

`core.target`

- `TicTargetKind`：目标类型，包含 `TOOL`、`ARMOR`、`UNKNOWN`。
- `TicTarget`：绑定 `ItemStack`、目标类型和 root NBT 访问。
- `TicTargets`：判断工具/盔甲、获取材料、解析盔甲槽位、列出已知 TiC/ConArm 物品。

`core.nbt`

- `TicNbt`：NBT 基础工具。
- 提供 compound、list、string list、numeric 字段的读取、复制、追加、删除和数值写回。
- 不包含 stats、fluid、trait 等业务语义。

`core.data`

- `TicDataAccess`：封装 `root.ticlib`。
- 负责 `DataVersion`、`DirtyVersion`、`Components`。
- 模块持久数据通过 `component(id)` 和 `writableComponent(id)` 读写。

`core.tconstruct`

- `TicNativeAccess`：封装 TiC 原生 NBT 操作。
- 统一处理 `Stats`、`Traits`、`Modifiers`、`Base.Modifiers`。
- trait 添加/移除、数值 stat 修改、base modifier 操作都从这里进入。

`core.lifecycle`

- `TicModule`：模块生命周期接口。
- `TicLifecycleBus`：模块注册、build、copy、replay、validate 调度。
- `TicLifecycleContext`：生命周期上下文，包含 root、kind、itemId、materials。
- `TicLifecycleEvents`：Forge/TiC/ConArm 事件入口。

### 3.2 module

`module` 是具体功能层。模块可以包含业务逻辑，但应通过 `core` 读写底层结构。

当前模块：

- `module.stats.PersistentStatsModule`
- `module.fluid.FluidModule`
- `module.armor.ArmorTraitCacheModule`
- `module.crafttweaker.CraftTweakerBuildModule`

模块之间尽量保持独立。跨模块共享能力时，优先通过 `api` 或 `core` 抽象完成。

### 3.3 api

`api` 是外部 Java 和 CT 集成层依赖的入口。

当前 facade：

- `TicItems`
- `TicTraits`
- `TicStats`
- `TicFluids`
- `TicArmor`

CraftTweaker 的 `TicTool`、`ItemStackExpansion`、`PlayerExpansion` 也通过这些 facade 访问底层能力。

## 4. 生命周期

入口在 `TicLib.preInit`。

当前注册内容：

```java
TicLifecycleBus.register(PersistentStatsModule.INSTANCE);
TicLifecycleBus.register(FluidModule.INSTANCE);
TicLifecycleBus.register(CraftTweakerBuildModule.INSTANCE);

MinecraftForge.EVENT_BUS.register(TicLifecycleEvents.INSTANCE);
MinecraftForge.EVENT_BUS.register(FluidEvents.INSTANCE);
MinecraftForge.EVENT_BUS.register(CraftTweakerBuildModule.INSTANCE);
MinecraftForge.EVENT_BUS.register(ArmorTraitCacheModule.INSTANCE);
```

核心流程：

1. TiC/ConArm build event 进入 `TicLifecycleEvents`。
2. `TicLifecycleBus.build` 按模块注册顺序调用 `onBuild`。
3. 工具修改时，把修改前 stack 的 `root.ticlib` 复制到修改后 stack。
4. 调用模块 `onCopy`。
5. 调用模块 `onReplay`。
6. 调用模块 `onValidate`。

模块职责划分：

- `onBuild`：参与当前 build root 的即时修改。
- `onCopy`：处理修改前后两个上下文之间的状态转移。
- `onReplay`：把 `Components` 中的持久状态重放到当前 root。
- `onValidate`：裁剪、清理或修正非法状态。

## 5. TiC 原生数据操作规范

适合操作 TiC 原生数据的场景：

- 添加或移除真实注册 trait。
- 修改当前 build 过程中的 stat。
- 写入 `Base.Modifiers`，交给 TiC 原生后续 rebuild 使用。
- 查询材料、trait、modifier、stats。

不适合作为 TiC 原生数据保存的内容：

- TicLib 模块私有状态。
- token 去重记录。
- 流体内容。
- 自定义模块内部进度。
- 运行时缓存。

这些内容应保存到：

```text
root.ticlib.Components.<moduleId>
```

## 6. 现有模块

### 6.1 PersistentStatsModule

组件 ID：

```text
ticlib:stats
```

职责：

- 保存 token 化 stat 增量。
- 对外提供统一 `addStat` 语义。
- 整数、浮点和其它 NBT 数值字段共用同一逻辑。
- replay 时只修改当前 `Stats` 中已存在的数值字段。

约束：

- 不创建未知 stat 字段。
- token 数据保存在 TicLib 组件中。
- 实际数值写回由 `TicNativeAccess` 和 `TicNbt.addNumeric` 完成。

### 6.2 FluidModule

组件 ID：

```text
ticlib:fluid
```

职责：

- 给 TiC 工具和 ConArm 盔甲挂 Forge `FLUID_HANDLER_ITEM_CAPABILITY`。
- 保存内置容量和当前流体。
- 支持 modifier 实现 `TicFluidTankProvider` 提供额外 tank。
- validate 时裁剪超过容量的流体，容量为 0 时清空流体。

约束：

- capability handler 每次从当前 stack NBT 读取数据。
- 默认不拦截物品右键方块。
- 世界交互由调用方显式使用 `TicFluids.interactWithFluidHandler`。

### 6.3 ArmorTraitCacheModule

职责：

- 缓存玩家当前四件盔甲的 trait 快照。
- 监听换装、登录、登出、clone。
- 提供快速查询 `hasTrait`、`hasSlotTrait`。

约束：

- 缓存只存在内存中。
- 不写 ItemStack NBT。
- 玩家装备变化后刷新。

### 6.4 CraftTweakerBuildModule

职责：

- 把 CT build event 纳入 lifecycle。
- 在 `onBuild` 中发布工具/盔甲构建事件。
- 脚本 reload 时清空 CT handler。

约束：

- CT 层只调用 `api.*`。
- CT build event 只操作当前 build root。
- 持久化状态通过模块组件完成。

## 7. 新功能接入流程

新增一个持久化功能时，推荐流程：

1. 在 `module.xxx` 下新增 `XxxModule implements TicModule`。
2. 定义组件 ID，例如 `ticlib:xxx` 或外部 mod 自己的 `modid:xxx`。
3. 用 `TicDataAccess.writableComponent(id)` 写入私有 NBT。
4. 在 `onReplay` 中把私有组件重放到当前 root 或 runtime 状态。
5. 在 `onValidate` 中清理非法或过期数据。
6. 在 `TicLib.preInit` 中注册模块。
7. 在 `api` 下新增或扩展 facade。
8. 如果需要 CT 暴露，让 CT 类只调用 `api.*`。
9. 如果需要 Forge capability 或 event，模块可以注册自己的 event handler，但数据仍走 `core`。

最小模块示例：

```java
public final class ExampleModule implements TicModule {
    public static final ExampleModule INSTANCE = new ExampleModule();
    public static final String ID = "ticlib:example";

    @Override
    public String id() {
        return ID;
    }

    @Override
    public boolean supports(TicTargetKind kind) {
        return kind == TicTargetKind.TOOL || kind == TicTargetKind.ARMOR;
    }

    @Override
    public void onReplay(TicLifecycleContext context) {
        NBTTagCompound data = context.data().component(ID);
        if (data == null) {
            return;
        }
        // 读取私有组件，重放到当前 root 或 runtime 状态。
    }
}
```

## 8. 开发规范

通用规范：

- `core` 保持小而稳定，不塞具体业务。
- `module` 负责业务逻辑，但通过 `core` 读写底层结构。
- `api` 是外部稳定入口。
- CT 集成层只依赖 `api`。
- 模块持久状态写入 `root.ticlib.Components`。
- 运行时缓存不写入 ItemStack NBT。
- 高频查询可以缓存，持久状态必须回到组件数据。

NBT 规范：

- root NBT 结构由 `TicDataAccess` 和 `TicNativeAccess` 统一处理。
- 不在 CT、event、module 中散写重复的 NBT 路径。
- 组件 ID 使用稳定字符串，例如 `ticlib:stats`。
- 组件内部字段使用首字母大写的 NBT key，例如 `Capacity`、`Fluid`、`Bonuses`。

TiC 原生数据规范：

- trait 相关逻辑走 `TicTraits` 或 `TicNativeAccess`。
- stat 相关逻辑走 `TicStats` 或 `TicNativeAccess`。
- base modifier 相关逻辑走 `TicTraits` 或 `TicNativeAccess`。
- 不直接在业务层手写 `Traits`、`Modifiers`、`Base.Modifiers` 的 list 操作。

模块设计规范：

- 一个模块只处理一个明确领域。
- 模块注册顺序需要能解释。
- `onReplay` 应可重复执行。
- `onValidate` 不应依赖外部调用顺序。
- capability provider 不缓存会随 rebuild 改变的 NBT 快照。
- 对外 API 返回 copy 或只读视图，避免外部直接改内部状态。

文档规范：

- 架构变化更新根目录 `说明.md`。
- Java/CT 用法变化更新 `docs/Crt.md`。
- 文档中只保留当前推荐入口。

## 9. 测试与验收

基础编译：

```powershell
.\gradlew.bat build
```

搜索检查：

```powershell
rg "TicToolStats|TicToolTraits|TicToolStacks|TicState|ToolFluid|addIntStat|withIntStat" src docs
```

期望：没有非预期引用。

建议游戏内 smoke test：

- 工具和盔甲都能被 `TicItems` 正确识别。
- `TicStats.add` 添加的属性在装配台 rebuild 后仍保留。
- 工具和盔甲设置流体容量后都有 Forge fluid capability。
- 流体容量降低后，多余流体会被裁剪。
- 玩家换盔甲后，`TicArmor.hasTrait` 查询结果刷新。
- CT build event 能在工具和盔甲构建时触发。

## 10. 快速定位

常用入口：

```text
src/main/java/com/smd/ticlib/TicLib.java
src/main/java/com/smd/ticlib/api
src/main/java/com/smd/ticlib/core
src/main/java/com/smd/ticlib/module
src/main/java/com/smd/ticlib/integration/crafttweaker
docs/Crt.md
```

判断一个改动应该放哪里：

- 是底层工具能力：放 `core`。
- 是具体功能：放 `module`。
- 是外部调用入口：放 `api`。
- 是 CT 适配：放 `integration.crafttweaker`。
- 是用户脚本用法：写入 `docs/Crt.md`。
- 是架构和开发规范：写入本文档。
